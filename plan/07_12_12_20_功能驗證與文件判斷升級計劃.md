# 功能驗證與文件判斷升級計劃

## 核心想法

`docs/architecture.md` 和 `docs/feature-map.md` 的價值，不是一次寫到完整，而是透過一條一條 feature 驗證，逐步確認它們是否真的能幫助人與 LLM agents 快速定位程式、縮小 context、避免誤改。

現在的觀念需要比原本 `07_12_00_33.md` 再升級一層：

- 每個 feature 都可以使用同一套驗證骨架。
- 但不同 feature 的驗證方式會不同。
- 有些 feature 還需要判斷是否要查 `docs/api.md` 或 `docs/data-format.md`。
- `feature-map.md` 的 feature name 不一定一對一等於 `architecture.md` 的 module name。
 
## 升級版驗證流程

針對每一個 feature，建議使用以下流程：

```text
1. architecture.md 能判斷 module 邊界
2. feature-map.md 能找到主要 entry points
3. Select-String 搜尋結果與 docs 一致
4. 判斷是否牽涉 docs/api.md
5. 判斷是否牽涉 docs/data-format.md
6. 依 feature 類型完成 Browser / API / data 驗證
7. 讀完相關程式後沒有明顯缺漏
8. docs 有需要補的地方已更新
```

這套流程的骨架固定，但第 4、5、6 步要依 feature 類型調整。

## 為什麼要加入 api.md 與 data-format.md 判斷

原本只看 `architecture.md` 和 `feature-map.md`，對 Search 這種單純 UI 行為通常夠用。

但當 feature 牽涉資料保存、匯入匯出、server endpoint、JSON shape 時，只靠 `feature-map.md` 會不夠。這些內容應該拆到專門文件：

```text
docs/api.md
docs/data-format.md
```

判斷規則：

```text
看到 server route / fetch endpoint，就查 api.md。
看到 JSON shape / state schema / import/export，就查 data-format.md。
```

## 何時需要查 docs/api.md

當 feature 牽涉以下內容時，需要查 `docs/api.md`：

- `fetch('/api/...')`
- `server.js`
- `GET /api/users`
- `GET /api/projects`
- `GET /api/flowchart`
- `POST /api/flowchart`
- request / response shape
- server file path rules
- user/project 儲存規則
- localStorage 與 server fallback 的優先順序
 
範例：

```text
Storage / API / Persistence -> 一定需要 api.md
Account / Project Switchers -> 可能需要 api.md，因為會讀 /api/users 與 /api/projects
Search Bar -> 通常不需要 api.md
Node Create / Edit / Render -> 通常不需要 api.md，除非驗證 save/reload
```

## 何時需要查 docs/data-format.md

當 feature 牽涉以下內容時，需要查 `docs/data-format.md`：

- `flowchart.json`
- `exportJSON()`
- `importJSON()`
- `nodes`
- `edges`
- `flows`
- `groups`
- `display`
- `view`
- `currentFlowId`
- `version`
- state schema
- 舊資料 fallback / migration

範例：

```text
Storage / API / Persistence -> 一定需要 data-format.md
Node Create / Edit / Render -> 大概率需要 data-format.md，因為會改 nodes[]
Edge Create / Edit / Route -> 大概率需要 data-format.md，因為會改 edges[]
Flow / Subflow Navigation -> 可能需要 data-format.md，因為會改 flows[] 與 currentFlowId
Search Bar -> 通常不需要 data-format.md，除非搜尋邏輯依賴資料 schema
```

## 不同 feature 的驗證方式不同

同一套流程可以套用到每個 feature，但第 6 步的驗證方式需要依功能調整。

### Search Bar

驗證重點：

- input keyword
- results count
- previous / next
- focus result
- status display

通常不需要：

- `docs/api.md`
- `docs/data-format.md`

### Storage / API / Persistence

驗證重點：

- `exportJSON()` / `importJSON()` shape
- localStorage fallback
- `GET /api/flowchart`
- `POST /api/flowchart`
- reload 後資料是否保存

通常需要：

- `docs/api.md`
- `docs/data-format.md`

### Account / Project Switchers

驗證重點：

- account menu
- project menu
- `/api/users`
- `/api/projects`
- current user/project label
- 切換前是否保存目前 project

通常需要：

- `docs/api.md`

可能需要：

- `docs/data-format.md`，如果驗證切換後載入的 project JSON。

### Flow / Subflow Navigation

驗證重點：

- breadcrumb
- Back button
- current flow
- subflow button
- empty subflow remove
- nodes / edges / groups 是否依 current flow 過濾

可能需要：

- `docs/data-format.md`，因為會牽涉 `flows[]`、`currentFlowId`、node `subflowId`。

### Node Create / Edit / Render

驗證重點：

- add node
- title/body editing
- tags
- shape picker
- resize / move
- connector points
- subflow button
- undo/save

大概率需要：

- `docs/data-format.md`，因為會牽涉 `nodes[]` 的資料格式。

可能需要：

- `docs/api.md`，如果驗證 save/reload 或 server persistence。

## module name 與 feature name 的關係

`architecture.md` 和 `feature-map.md` 的命名角度不同。

```text
architecture.md = 系統責任邊界
feature-map.md = 使用者功能入口地圖
```

因此 feature name 不一定一對一等於 module name。

範例：

```text
Search Bar -> Search module
Account / Project Switchers -> Account / Project module
Storage / API / Persistence -> Storage / API module
Node Create / Edit / Render -> 主要對應 Node Renderer，但也牽涉 Selection / Dragging、Flow / Subflow、Edge Router / Renderer、Storage / API
```

如果 feature 牽涉多個 modules，建議在驗證時明確寫：

```text
Primary module:
- Node Renderer

Related modules:
- Selection / Dragging
- Flow / Subflow
- Edge Router / Renderer
- Storage / API
```

## 建議驗證順序

優先從常改、範圍清楚、風險可控的功能開始：

1. `Search Bar`
2. `Account / Project Switchers`
3. `Flow / Subflow Navigation`
4. `Storage / API / Persistence`
5. `Node Create / Edit / Render`
6. `Edge Create / Edit / Route`
7. `Import / Export Drawer`
8. `Toolbar / Display Controls`

理由：

- `Search Bar` 範圍小，適合校準 docs 顆粒度。
- `Account / Project Switchers` 能練習 API 與 UI menu 驗證。
- `Flow / Subflow Navigation` 能理解 flow scope。
- `Storage / API / Persistence` 會帶出 `api.md` 與 `data-format.md`。
- `Node` 與 `Edge` 較複雜，適合在驗證流程穩定後再做。

## 驗證時要檢查什麼

每個 feature 都確認：

- `architecture.md` 是否能判斷 module responsibility。
- `feature-map.md` 是否列出足夠的 HTML / CSS / JS / state / API entry points。
- `Select-String` 搜尋結果是否與 docs 一致。
- 是否需要查 `docs/api.md`。
- 是否需要查 `docs/data-format.md`。
- `Verify` 步驟是否符合該 feature 的真實風險。
- 實際 code 與 docs 是否一致。
- 是否有跨 module dependency 沒有記錄。

如果 docs 讓人或 LLM 還是必須全文搜尋很多次，代表 docs 太粗，需要補。

如果 docs 寫了大量 function 內部細節，代表 docs 太細，需要收斂成導航資訊。

## 可直接使用的提示詞

```text
請使用升級版驗證流程驗證 feature-map.md 的「功能名稱」。
此 feature 對應 architecture.md 的「Module 名稱」。

請依序確認：
1. architecture.md 能判斷 module 邊界
2. feature-map.md 能找到主要 entry points
3. Select-String 搜尋結果與 docs 一致
4. 判斷是否牽涉 docs/api.md
5. 判斷是否牽涉 docs/data-format.md
6. 依 feature 類型完成 Browser / API / data 驗證
7. 讀完相關程式後整理簡短描述
8. 如果 docs 有缺漏，提出需要補的項目
```

## 執行策略

不要一次補完所有文件。每次選一條 feature 驗證，讓文件從實際理解中長出來。

每次驗證後都問：

```text
這次有沒有新發現的 entry point？
這次有沒有牽涉 API？
這次有沒有牽涉資料格式？
這次是否需要補 feature-map.md？
這次是否需要補 architecture.md？
這次是否需要補 api.md 或 data-format.md？
```

這樣文件會保持輕、準、可導航，也會讓專案理解逐步累積。
