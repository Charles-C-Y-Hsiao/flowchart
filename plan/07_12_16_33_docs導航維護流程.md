# Docs 導航文件維護流程

## 背景

專案目前建立 `docs/architecture.md`、`docs/feature-map.md`、`docs/data-format.md`、`docs/api.md`，用來幫助人與 LLM agents 在修改功能時快速定位 context。

這些文件不應該寫成完整教學，而是作為導航工具：

- `architecture.md`：從 module ownership 角度判斷邊界、state ownership、dependency direction。
- `feature-map.md`：從 user-facing feature 角度找到 HTML / CSS / JS / state / API / data entry points。
- `data-format.md`：當 feature 牽涉 JSON schema、state serialization、import/export 時使用。
- `api.md`：當 feature 牽涉 server route、request / response、persistence 時使用。

## 核心原則：用 symbol 找入口，不要相信行號永遠正確

`feature-map.md` 可以保留 line number，因為它能幫助第一次閱讀時快速跳到附近；但 line number 不是穩定 contract。

真正穩定的導航入口應該是：

- function name，例如 `groupBounds()`、`renderGroup()`、`saveState()`。
- CSS class，例如 `.node-group`、`.flow-search-button`。
- DOM id，例如 `#btnAddLane`、`#btnSearchPrev`。
- state key，例如 `state.groups`、`state.search`。
- API route，例如 `GET /api/flowchart`。
- data field，例如 `groups.nodeIds`、`nodes.subflowId`。

也就是說，改程式後不需要因為前面新增幾行，就重算整份 `feature-map.md` 的所有 line numbers。

比較好的檢查方式是只針對被修改的 feature，用 `Select-String` 找 symbol：

```powershell
Select-String -Path .\flowchart.html -Pattern 'groupBounds|node-group|renderGroup|state.groups'
Select-String -Path .\flowchart.html -Pattern 'renderNode|node-subflow|state.nodes|subflowId'
```

如果 symbol 還存在、責任沒有改、入口仍能帶到正確程式，line number 漂移通常不需要更新。

需要更新 docs 的情況是：

- function / class / DOM id / state key 改名。
- feature 被拆到新檔案。
- module ownership 改變。
- feature 新增或移除重要依賴。
- entry point、verify steps、API route、data format 改變。

不需要更新 docs 的情況是：

- 只是同一個 function 內部微調。
- 只是前面新增幾行導致 line number 漂移。
- 只是文字、樣式數值、小型 bug fix，且入口 symbol 沒變。

## 文件語言規則

- 程式相關名稱使用英文，例如 `Module`、`State`、`Entry points`、`Verify`、`Search Bar`、`state.search`、`flowchart.html`。
- 說明文字使用繁體中文，讓文件更貼近日常維護語境。
- Code symbols、file paths、CSS classes、DOM ids、function names 保持原樣。
- 不複製過多 implementation details，只記錄未來定位功能時需要知道的資訊。

## `feature-map.md` 建議固定格式

每個 feature 建議使用下面格式：

```md
## Feature Name

使用者介面元素：

- ...

對應 modules：

- 主要 module：`Primary Module`
- 相關 modules：`Related Module A`, `Related Module B`

目前程式入口：

- CSS / HTML / DOM / state / function / API / data entry points。

驗證方式：

- Browser / API / data / Select-String 驗證步驟。
- 不建議從哪裡開始處理。
```

## 為什麼要加「對應 modules」

`feature-map.md` 是從「功能」角度切；`architecture.md` 是從「module ownership」角度切。

像 `Node Renderer`、`Group Renderer` 這類 feature，本來就不一定只對應一個 module。它們可能跨到互動、資料格式、流程範圍、儲存、edge redraw。

範例：

```md
對應 modules：

- 主要 module：`Node Renderer`
- 相關 modules：`Selection / Dragging`, `Flow / Subflow`, `Edge Router / Renderer`, `Storage / API`, `Display / Theme`
```

```md
對應 modules：

- 主要 module：`Group Renderer`
- 相關 modules：`Selection / Dragging`, `Flow / Subflow`, `Node Renderer`, `Edge Router / Renderer`, `Storage / API`, `Import / Export`
```

這樣 LLM 讀 feature 時會先知道：

- 主要應該修改哪個 module。
- 哪些周邊 module 只需要理解或驗證。
- 哪些地方不應該直接亂改。

## 每次 LLM 改完程式後的順手流程

可以把這當成固定收工檢查：

```text
改功能前：
1. 先讀 docs/architecture.md 判斷 module ownership。
2. 再讀 docs/feature-map.md 找 feature entry points。
3. 如果牽涉 API，讀 docs/api.md。
4. 如果牽涉 JSON / storage / import / export，讀 docs/data-format.md。
5. 用 Select-String 搜 symbol，而不是只依賴 line number。
6. 只載入相關 context 開始改。

改功能後：
7. 確認這次是否改到 id / class / function / state / dependency / API route / data field。
8. 如果 entry point、verify steps 或 user-facing control 改變，更新 docs/feature-map.md。
9. 如果 module ownership、state ownership 或 dependency direction 改變，更新 docs/architecture.md。
10. 如果資料格式改變，更新 docs/data-format.md。
11. 如果 API 行為改變，更新 docs/api.md。
12. Final handoff 說明 docs 已同步，或說明這次沒有 navigation boundary 變更所以不需更新。
```

## 可直接使用的提示詞

```text
請依照 docs/architecture.md 和 docs/feature-map.md 修改這個功能。

修改前：
- 先判斷主要 module 與相關 modules。
- 使用 Select-String 搜尋 function name / CSS class / DOM id / state key，不要只依賴 line number。
- 如果牽涉 API，讀 docs/api.md。
- 如果牽涉資料格式、storage、import/export，讀 docs/data-format.md。

完成後：
- 如果入口、state、模組責任、依賴、API 或資料格式有變，請同步更新相關 docs。
- 如果不用更新，請在最後說明原因。
```

## 目標

這個流程的目標不是讓每次修改都變成文件工作，而是讓文件保持輕、準、可導航。

只要新的改動會影響下一次「從哪裡開始找」，就更新 docs；如果只是 function 內部細節，通常不需要更新。

判斷重點不是「行號有沒有漂移」，而是「下一次人或 LLM 能不能靠 symbol 快速找到正確入口」。
