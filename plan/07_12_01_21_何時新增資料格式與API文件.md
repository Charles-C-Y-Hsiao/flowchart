# 何時新增資料格式與 API 文件

## 背景

在驗證 `Storage / API / Persistence` 時，會很自然地遇到兩種資訊：

1. 資料本身長什麼樣子。
2. 前後端如何透過 API 讀寫資料。

這兩種資訊如果全部塞進 `docs/feature-map.md`，文件會變得太肥，失去「快速導航」的用途。因此需要分拆成：

```text
docs/data-format.md
docs/api.md
```

## 判斷規則

一句話版本：

```text
看到 JSON shape，就補 data-format.md。
看到 server route / fetch endpoint，就補 api.md。
```

## 什麼時候需要 data-format.md

當你在 feature 驗證中看到以下內容，就表示需要資料格式文件：

- `flowchart.json`
- `exportJSON()`
- `importJSON()`
- `version`
- `currentFlowId`
- `view`
- `display`
- `flows`
- `groups`
- `nodes`
- `edges`
- 舊資料 fallback / migration

這些不是某一個 UI 功能的小細節，而是整個專案的資料契約。

`docs/data-format.md` 應該回答：

- JSON top-level fields 有哪些。
- 每個主要 collection 的資料 shape。
- 哪些欄位是必要欄位。
- 哪些欄位有 fallback。
- 修改資料格式時要檢查哪些 function。

## 什麼時候需要 api.md

當你在 feature 驗證中看到以下內容，就表示需要 API 文件：

- `server.js`
- `fetch('/api/...')`
- `GET /api/users`
- `GET /api/projects`
- `GET /api/flowchart`
- `POST /api/flowchart`
- request / response shape
- server file path rules
- user/project slug rules

這些是前後端契約，不應該只藏在 `feature-map.md` 的 Storage 區塊中。

`docs/api.md` 應該回答：

- Server entry 在哪裡。
- 有哪些 endpoints。
- 每個 endpoint 的 request / response shape。
- 前端哪些 function 會呼叫它。
- server 讀寫哪些檔案。
- API 失敗時前端怎麼 fallback。

## 與 feature-map.md 的關係

`feature-map.md` 只需要保留「入口」：

```text
- Server save/load：saveFlowchart() / readFlowchart()
- API routes：GET /api/users, GET /api/projects, GET/POST /api/flowchart
- 詳細資料格式：docs/data-format.md
- 詳細 API 說明：docs/api.md
```

不要把完整 JSON schema 或所有 endpoint 細節塞進 `feature-map.md`。

## 新手如何意識到要拆文件

可以用這三個問題判斷：

1. 這些資訊是不是很多 feature 都會共用？
2. 這些資訊是不是牽涉資料相容性或前後端契約？
3. 如果放在 feature-map.md，會不會讓單一 feature 區塊變得很長？

如果答案是 yes，就應該拆成獨立文件。

## 本次實際案例

驗證 `Storage / API / Persistence` 時，發現：

- `feature-map.md` 有前端 storage 入口。
- 但 `server.js` 的 API routes 原本沒有被清楚列入。
- 讀程式時額外找到 `saveFlowchart()`、`readFlowchart()`、`listUsers()`、`listProjects()` 與 `/api/...` routes。

因此本次新增：

```text
docs/data-format.md
docs/api.md
```

並同步補強 `docs/feature-map.md` 的 `Storage / API / Persistence` 區塊，讓它能導航到 server 入口與新文件。
