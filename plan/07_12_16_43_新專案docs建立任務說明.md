# 新專案建立 docs 導航文件的 LLM 任務說明

## 使用目的

當我要在一個新專案中建立 docs 導航文件時，可以把這份文件交給 LLM。

目標不是產生一般介紹文件，而是建立能幫助人與 LLM agents 快速定位功能、縮小 context、避免亂改程式的導航文件。

## 任務目標

請在新專案中建立或評估是否需要建立：

```text
docs/architecture.md
docs/feature-map.md
docs/data-format.md
docs/api.md
```

四份文件的目的如下：

- `docs/architecture.md`：描述 module 邊界、state ownership、data flow、dependency direction、LLM 修改程式時的 context rules。
- `docs/feature-map.md`：把 user-facing features 對應到目前 code entry points，包含 HTML / CSS / JS / state / API / data / verify steps。
- `docs/data-format.md`：描述主要資料格式、JSON schema、storage payload、import/export 欄位、相容性規則。
- `docs/api.md`：描述 server routes、request / response、query params、persistence 行為、只讀驗證方式。

如果新專案沒有 server/API，可以先建立簡短的 `docs/api.md` 說明「目前沒有 API」；如果沒有資料持久化，也可以在 `docs/data-format.md` 說明目前沒有穩定資料格式。重點是讓後續 LLM 知道它不用猜。

## 重要原則

1. 文件是導航工具，不是完整教學。
2. 不要一次追求完美，先建立可用第一版。
3. 程式相關名稱保留原文，例如 file path、function name、CSS class、DOM id、state name、API endpoint。
4. 說明文字使用繁體中文。
5. 如果專案目前是單檔大檔案，也先忠實記錄現況，不必急著拆檔。
6. `feature-map.md` 可以保留 line number，因為它能幫助第一次閱讀時快速跳到附近；但 line number 不是穩定 contract。
7. 真正穩定的導航入口是 symbol：function name、CSS class、DOM id、state key、API route、data field。
8. 不要複製大量 implementation details，只記錄未來改功能時需要知道的入口、責任、依賴與驗證方式。

## 搜尋方式：用 symbol 找入口

如果在 PowerShell 環境，優先使用：

```powershell
Select-String -Path .\目標檔案 -Pattern 'keyword|functionName|className|stateName'
```

範例：

```powershell
Select-String -Path .\flowchart.html -Pattern 'groupBounds|node-group|renderGroup|state.groups'
Select-String -Path .\server.js -Pattern 'GET /api|POST /api|saveFlowchart|readFlowchart'
Select-String -Path .\docs\*.md -Pattern 'Node Renderer|Group Renderer|Storage / API'
```

如果環境可使用 `rg`，也可以用：

```powershell
rg -n "keyword|functionName|className|stateName"
```

但如果文件主要給 Windows PowerShell 使用者，請以 `Select-String` 範例為主。

## 建議工作流程

請依照以下順序進行：

```text
1. 掃描專案檔案結構。
2. 找出主要 entry files，例如 HTML、frontend entry、server entry、config、data files。
3. 使用搜尋工具找出主要 features、functions、event bindings、API routes、state definitions、data fields。
4. 建立 docs/architecture.md，描述 module map、data flow、state ownership、context rules。
5. 建立 docs/feature-map.md，從 user-facing features 對應到 code entry points。
6. 若專案有 JSON / storage / import / export / persisted state，建立 docs/data-format.md。
7. 若專案有 server routes / API calls / persistence endpoints，建立 docs/api.md。
8. 用一個最小功能做驗證，確認 docs 能帶人找到正確程式。
9. 如果 docs 太粗或漏掉入口，補 feature-map.md。
10. 如果 module ownership 或 dependency direction 不準，補 architecture.md。
11. 如果資料欄位、API 行為需要解釋，補 data-format.md 或 api.md。
```

## `architecture.md` 建議內容

`docs/architecture.md` 建議包含：

```text
# 專案 Architecture

## 目的
說明這份文件是導航文件，不是完整介紹文件。

## 目前結構
列出主要檔案、主要資料夾、前端入口、後端入口、資料檔。

## Module Map
用表格整理：
- Module
- Responsibility
- Owns / Writes
- Reads / Depends On
- Do Not Touch By Default

## Data Flow
描述 user event、state、render、storage、API 之間的流向。

## State Ownership
列出主要 state / data model 由誰擁有、誰可以修改。

## Context Rules For LLM Work
說明 LLM 修改功能時如何縮小 context：
先判斷 module -> 查 feature-map.md -> 必要時查 data-format.md / api.md -> 只讀 entry points -> 有真實依賴再擴大。

## Future Split Direction
如果專案目前是大檔案，列出未來可能拆分方向，但不要要求一次完成。

## Documentation Update Workflow
說明每次修改功能後，什麼情況要同步更新 docs。
```

## `feature-map.md` 建議內容

`docs/feature-map.md` 建議用 user-facing feature 當章節，而不是用檔案名稱當章節。

每個 feature 建議使用下面格式：

```md
## Feature Name

使用者介面元素：

- DOM id / button / input / route / API endpoint。

對應 modules：

- 主要 module：`Primary Module`
- 相關 modules：`Related Module A`, `Related Module B`

目前程式入口：

- CSS 樣式：`path/to/file` 或 `.css-class`
- HTML / JSX / Template：`path/to/file`
- DOM 參照或 component：`functionName()` / `ComponentName`
- State 狀態：`stateName`
- 主要 action / handler：`functionName()`
- API / Storage：`endpoint` / `storageKey`
- Data format：`fieldName` / `docs/data-format.md`
- Event binding：`path/to/file`

驗證方式：

- Select-String 搜尋哪些 symbols。
- Browser / API / data 驗證步驟。
- 預期結果。

不建議從這裡開始處理：

- 與此 feature 無關的 module 或檔案。
```

### `feature-map.md` 的 line number 規則

可以寫：

```md
- Group render：`renderGroup()` around `flowchart.html:4677`
- CSS 樣式：`.node-group` around `flowchart.html:1310`
```

但請在文件開頭明確說明：

```md
Line number 只是一個附近提示，不是穩定 contract。
維護時請優先用 function name、CSS class、DOM id、state key、API route、data field 找入口。
如果只是前面新增幾行造成 line number 漂移，不需要重算整份文件。
```

## `data-format.md` 何時需要建立或更新

如果專案有下列任一情況，請建立或更新 `docs/data-format.md`：

- 有 JSON data file。
- 有 localStorage / IndexedDB / database payload。
- 有 import/export。
- 有 server persistence。
- 有前端 state 會被 serialize / deserialize。
- 修改 feature 時新增、刪除或改名資料欄位。
- 需要說明 backward compatibility / default value / migration。

`docs/data-format.md` 建議包含：

```text
# Data Format

## 目的
說明這份文件記錄穩定資料格式，不是所有 runtime state。

## Data Sources
- 檔案位置 / storage key / DB table / API payload。

## Top-level Fields
- field
- type
- meaning
- owner module

## Feature-specific Sections
- nodes / edges / groups / users / projects / settings 等。

## Compatibility Notes
- 缺省值。
- 舊版本 fallback。
- migration 注意事項。

## When To Update
- 新增欄位。
- 刪除欄位。
- 改變欄位語意。
- import/export 行為改變。
```

## `api.md` 何時需要建立或更新

如果專案有下列任一情況，請建立或更新 `docs/api.md`：

- 有 server route。
- 前端使用 `fetch()` / axios / RPC。
- 有 request / response schema。
- 有 query params / path params。
- 有 persistence endpoint。
- 修改 feature 時 API endpoint、payload、status code、fallback 行為改變。

`docs/api.md` 建議包含：

```text
# API

## 目的
說明這份文件記錄可被前端或外部工具呼叫的 API。

## Base URL / Server Entry
- server entry file。
- base path。

## Endpoints
每個 endpoint 包含：
- Method + path
- Purpose
- Query params / body
- Response shape
- Side effects
- Frontend callers
- Verification command

## Persistence Behavior
- 哪些 endpoint 會讀檔。
- 哪些 endpoint 會寫檔。
- fallback / error handling。

## When To Update
- 新增 / 移除 route。
- request / response 改變。
- persistence 行為改變。
- frontend caller 改變。
```

## 文件語言規則

請使用這種混合風格：

- 說明文字：繁體中文。
- 程式符號：保持原樣。
- 表格欄位若是常用架構詞可以保留英文，例如 `Module`、`Responsibility`、`State`、`Entry points`。
- Feature 標題可保留英文，例如 `Search Bar`、`Storage / API / Persistence`，方便和程式命名對應。

範例：

```md
目前程式入口：

- CSS 樣式：`.flow-search-button` around `flowchart.html:1407`
- DOM 參照：`btnSearchPrev`, `btnSearchNext`
- State 狀態：`state.search`
- 搜尋結果建立：`buildProjectSearchResults()`
- Event binding：search input and prev/next buttons
```

## 第一版驗證方式

建立完 docs 後，不要只停在文件產出。請選一個小功能驗證：

```text
1. 從 architecture.md 判斷 module。
2. 從 feature-map.md 找 entry points。
3. 用 Select-String 驗證 symbol 是否存在。
4. 判斷是否需要讀 data-format.md。
5. 判斷是否需要讀 api.md。
6. 實際讀相關程式碼。
7. 如可執行，操作 UI、跑 API GET、或跑測試驗證。
8. 如果文件有漏，更新 docs。
```

## 可直接交給 LLM 的任務提示詞

```text
請幫我在這個新專案建立 docs 導航文件：

- docs/architecture.md
- docs/feature-map.md
- docs/data-format.md
- docs/api.md

請注意：

1. 這些文件是給人與 LLM agents 維護程式時快速定位 context 用，不是一般介紹文件。
2. 說明文字使用繁體中文；程式符號、file paths、function names、CSS classes、DOM ids、state keys、API routes 保持原樣。
3. architecture.md 請從 module ownership 角度整理。
4. feature-map.md 請從 user-facing feature 角度整理，並為每個 feature 加上：
   - 使用者介面元素
   - 對應 modules
   - 目前程式入口
   - 驗證方式
   - 不建議從哪裡開始處理
5. feature-map.md 可以保留 line number 幫助第一次定位，但 line number 不是穩定 contract。請優先使用 function name、CSS class、DOM id、state key、API route、data field 作為導航 symbol。
6. 如果專案有 JSON / storage / import / export / persisted state，請建立 data-format.md；如果沒有，也請簡短說明目前沒有穩定資料格式。
7. 如果專案有 server routes / frontend API calls，請建立 api.md；如果沒有，也請簡短說明目前沒有 API。
8. 建立完後，請選一個小 feature 做第一輪驗證：architecture -> feature-map -> Select-String symbols -> data-format/api 判斷 -> 讀程式 -> Browser/API/test 驗證 -> 補 docs。
9. 完成時請列出建立了哪些 docs、涵蓋哪些 modules/features、用哪個 feature 驗證，以及哪些地方需要後續補強。
```

## 完成回報格式

完成時請回報：

```text
已建立：
- docs/architecture.md
- docs/feature-map.md
- docs/data-format.md
- docs/api.md

已涵蓋的主要 modules：
- ...

已涵蓋的主要 features：
- ...

已用哪個 feature 做第一輪驗證：
- ...

data-format.md 判斷：
- 已建立 / 不需要，原因是 ...

api.md 判斷：
- 已建立 / 不需要，原因是 ...

需要後續補充：
- tests / AGENTS.md / more feature verification 等
```
