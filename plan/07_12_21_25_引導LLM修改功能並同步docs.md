# 引導 LLM 修改功能並同步 docs

## 目的

當我要請 LLM 修改某個功能時，不要只說「幫我改 X」，而是先引導它讀專案導航文件，讓它知道：

- 這次功能屬於哪個主要 module。
- 會牽涉哪些相關 modules。
- 應該從哪些 HTML / CSS / JS / state / API / data entry points 開始看。
- 哪些地方預設不應該碰。
- 改完後哪些 docs 需要同步更新。

`docs/architecture.md` 適合放穩定的 module ownership 規則；這份 plan 則適合放「可以直接貼給 LLM 的任務提示詞」。

## 建議使用方式

每次要改功能時，可以先貼下面這段給 LLM。

```text
我要修改 [Feature Name / 具體需求]。

請先不要直接改程式，先依照專案 docs 判斷修改範圍：

1. 先讀 `docs/architecture.md`
   - 判斷這次修改的主要 module。
   - 判斷相關 modules。
   - 確認 owns / writes 哪些 state。
   - 確認 reads / depends on 哪些資料或 module。
   - 列出預設不應該碰的區域。

2. 再讀 `docs/feature-map.md`
   - 找到對應 feature 章節。
   - 列出 HTML / CSS / DOM refs / state / functions / API / data-format entry points。
   - 注意：line number 只當附近位置提示，請以 symbol、id、class、function name、state key 為主。

3. 用 PowerShell `Select-String` 驗證 docs 和程式是否一致：
   ```powershell
   Select-String -Path .\flowchart.html -Pattern 'featureKeyword|functionName|className|idName|stateName'
   ```

   如果牽涉 server/API：
   ```powershell
   Select-String -Path .\server.js -Pattern 'apiRoute|functionName|project|user'
   ```

   如果牽涉 docs/data/API contract：
   ```powershell
   Select-String -Path .\docs\*.md -Pattern 'Feature Name|Module Name|fieldName|apiRoute'
   ```

4. 先回報你的判斷：
   - 主要 module：
   - 相關 modules：
   - 預計修改的檔案：
   - 預計修改的 symbols / ids / classes：
   - 不打算碰的區域：
   - 需要同步檢查的 docs：

5. 等判斷完成後再開始修改。

6. 修改完成後，請同步檢查 docs：
   - 若 feature entry point 改變，更新 `docs/feature-map.md`。
   - 若 module ownership / state ownership / dependency boundary 改變，更新 `docs/architecture.md`。
   - 若 `flowchart.json` 欄位或 import/export payload 改變，更新 `docs/data-format.md`。
   - 若 API endpoint / request / response / server persistence 改變，更新 `docs/api.md`。
   - 若完成 feature 驗證，更新 `docs/verification-status.md`。

7. 最後回報：
   - 程式改了什麼。
   - 驗證了什麼。
   - 更新了哪些 docs。
   - 哪些 docs 檢查後不需要更新，以及原因。
```

## 如果我只想讓 LLM 先規劃，不要改程式

可以改貼這段：

```text
我要修改 [Feature Name / 具體需求]。

請只做規劃，不要改程式。

請先讀：

1. `docs/architecture.md`
2. `docs/feature-map.md`
3. 如牽涉資料格式，再讀 `docs/data-format.md`
4. 如牽涉 API，再讀 `docs/api.md`
5. 如要看目前驗證狀態，再讀 `docs/verification-status.md`

請回報：

- 主要 module。
- 相關 modules。
- 應該從哪些 symbols / ids / classes / state keys 開始看。
- 需要用哪些 `Select-String` 指令驗證。
- 預計會修改哪些區域。
- 哪些區域預設不該碰。
- 改完後可能需要同步更新哪些 docs。
```

## 如果我已經改完功能，只要 LLM 幫我更新 docs

可以貼這段：

```text
我已經修改完 [Feature Name / Change Summary]。

請不要再改功能程式，只檢查 docs 是否需要同步。

請依序檢查：

1. `docs/architecture.md`
2. `docs/feature-map.md`
3. `docs/data-format.md`
4. `docs/api.md`
5. `docs/verification-status.md`

請用 `Select-String` 驗證 docs 內列出的 entry points 是否仍能在程式中找到。

只在以下情況更新 docs：

- feature entry point 改變。
- module ownership / dependency 改變。
- state shape / data-format 改變。
- API contract 改變。
- feature 驗證狀態需要更新。

不要只因為 line number 漂移就重寫 docs；line number 不是穩定 contract。
```

## 一句話原則

修改功能前，先讓 LLM 讀 `architecture.md` 判斷「該碰哪個 module」；再讀 `feature-map.md` 判斷「實際從哪裡改」；改完後才看是否需要同步 `data-format.md`、`api.md`、`verification-status.md`。
