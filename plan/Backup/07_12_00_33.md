# Docs 驗證建議與後續文件化計劃

## 核心想法

`docs/architecture.md` 和 `docs/feature-map.md` 不應該一次寫到看起來很完整，而是要透過一條一條功能來驗證是否真的好用。

每次驗證一個功能時，同時完成三件事：

1. 確認 docs 是否能幫助快速定位程式入口。
2. 補齊或修正 docs 中過少、過時或太籠統的內容。
3. 順手理解該功能在整個專案中的運作方式。

## 建議驗證流程

針對每一個 feature，使用以下流程：

```text
選一個功能
  -> 看 docs/architecture.md 判斷 module
  -> 看 docs/feature-map.md 找 entry points
  -> 用 rg 驗證 entry points 是否正確
  -> 實際閱讀相關程式碼
  -> 在 browser 操作驗證功能
  -> 補 docs 過少、過時或不清楚的地方
```

這樣 docs 會從真實使用中變準，而不是只停留在規劃層。

## 建議驗證順序

優先從常改、範圍清楚、風險可控的功能開始：

1. `Search Bar`
2. `Flow / Subflow Navigation`
3. `Node Create / Edit / Render`
4. `Edge Create / Edit / Route`
5. `Storage / API / Persistence`
6. `Import / Export Drawer`
7. `Toolbar / Display Controls`
8. `Account / Project Switchers`

理由：

- `Search Bar` 範圍小，適合校準 docs 的顆粒度。
- `Flow / Subflow Navigation` 可以理解 flow scope 與 breadcrumb。
- `Node` 是主要資料與 UI 核心。
- `Edge` 較複雜，適合等 docs 流程穩定後再驗證。
- `Storage / API` 需要小心，因為會影響資料保存與舊資料相容性。

## 驗證時要檢查什麼

每個 feature 都確認：

- `architecture.md` 是否能判斷 module responsibility。
- `feature-map.md` 是否列出足夠的 HTML / CSS / JS / state entry points。
- `Do Not Touch By Default` 是否真的能排除不相關 context。
- `Verify` 步驟是否足以確認功能沒有壞。
- 實際 code 與 docs 是否一致。
- 是否有跨 module dependency 沒有記錄。

如果 docs 讓人或 LLM 還是必須全文搜尋很多次，代表 docs 太粗，需要補。

如果 docs 寫了大量 function 內部細節，代表 docs 太細，需要收斂成導航資訊。

## flowchart.json 是否需要整理

需要，但它的角色不是一般 user-facing feature，而是資料格式與 state snapshot。

建議未來新增：

```text
docs/data-format.md
```

內容可包含：

- `flowchart.json` 的 top-level fields。
- `nodes` / `edges` / `flows` / `groups` / `display` 的資料格式。
- 哪些欄位是相容性關鍵。
- `exportJSON()` 與 `importJSON()` 如何處理資料。
- 舊資料格式的 fallback 或 migration 規則。

建議時機：

- 驗證 `Storage / API / Persistence` 時順手建立。
- 或第一次修改 state schema / import/export 格式時建立。

## server.js 是否需要整理

需要，但應該整理成 backend / API layer 文件，不要混進前端 feature map 太多。

建議未來新增：

```text
docs/api.md
```

內容可包含：

- `server.js` 提供哪些 endpoints。
- `/api/users`
- `/api/projects`
- `/api/flowchart`
- 檔案存放位置。
- `user` / `project` 參數規則。
- 前端哪些 function 會呼叫這些 API。
- Server 錯誤時前端 fallback 到哪裡。

建議時機：

- 驗證 `Storage / API / Persistence` 時建立。
- 或第一次修改 `server.js` / API route / project storage 行為時建立。

## 建議的 docs 結構

目前先維持：

```text
docs/
  architecture.md
  feature-map.md
```

當驗證到 storage 與 API 相關功能時，再補：

```text
docs/
  architecture.md
  feature-map.md
  data-format.md
  api.md
```

`architecture.md` 中的 `Storage / API` module 之後可以指向 `docs/data-format.md` 與 `docs/api.md`。

## 執行策略

不要一次補完所有文件。每次選一條功能驗證，讓文件從實際理解中長出來。

建議第一輪先做：

1. 用 `Search Bar` 驗證 `architecture.md` 與 `feature-map.md` 是否好用。
2. 修正 Search 相關 docs 顆粒度。
3. 再做 `Storage / API / Persistence`。
4. 做 Storage 時建立 `docs/data-format.md`。
5. 做 API 時建立 `docs/api.md`。

這樣文件會保持輕、準、可導航，也會讓專案理解逐步累積。
