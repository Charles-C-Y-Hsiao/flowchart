# Flowchart HTML 重構計劃

## 背景

目前 `flowchart.html` 已經接近 6000 行，CSS、HTML、JavaScript 全部集中在同一個檔案裡。這讓 LLM 可以很快幫忙加功能，但專案邊界逐漸變模糊，後續修改單一功能時，常常需要參考大量不相關程式碼，維護成本與誤改風險都會上升。

這份計劃的核心不是停止使用 LLM，而是讓專案結構能承受 LLM 的開發速度。目標是讓每次修改都有明確範圍，例如改搜尋功能只需要看搜尋模組，改節點樣式只需要看節點樣式與節點渲染，而不是每次都重新理解整份 `flowchart.html`。

## 目前觀察

- `flowchart.html` 大約 5987 行。
- CSS 區塊集中在檔案前段，約 `16-1826` 行。
- HTML 結構集中在中段，約 `1828-2008` 行。
- JavaScript 集中在後段，約 `2009-6056` 行。
- JavaScript 同時包含狀態管理、DOM 查找、搜尋、流程切換、節點渲染、連線演算法、拖拉事件、匯入匯出、API/localStorage、toolbar、使用者與專案管理。

目前的主要風險是功能邊界不清楚。當要修改一個小功能時，容易牽動大量不相關區塊，LLM 也容易因為上下文太大而加入多餘程式或產生不一致的寫法。

## 重構原則

1. 不做一次性大搬家。
2. 不優先追求漂亮架構，先追求可維護與可回退。
3. 每次只拆一個功能區塊，拆完就驗證。
4. 先拆低風險區塊，再拆核心行為。
5. 保留目前可運作的頁面，不改變資料格式與使用者流程。
6. 讓未來 LLM 能依照模組邊界工作，避免每次改動都讀整份檔案。

## 第一階段：建立專案地圖

新增一份架構文件，例如：

```text
docs/refactor-map.md
```

內容記錄目前 `flowchart.html` 的主要功能區塊：

- App state：`state`、current user/project、display settings。
- Storage/API：`loadSavedState`、`saveProjectFile`、localStorage、fetch。
- Flow/subflow：flow path、breadcrumb、current flow。
- Search：project search、prev/next、focus result。
- Node UI：render node、tag editor、shape picker、resize/edit。
- Edge routing：connector points、orthogonal route、redraw edges。
- Selection/dragging：marquee、multi-select、move/resize。
- Import/export：JSON、Mermaid、prompt guide。
- Toolbar/account/project UI。

這一步的目標是先讓專案有地圖。未來改功能時，可以先定位到對應區塊，再決定是否抽出模組。

## 第二階段：先拆 CSS

CSS 是低風險、高收益的第一刀。建議將 `<style>` 內容逐步拆成：

```text
assets/css/base.css
assets/css/toolbar.css
assets/css/flow-nav.css
assets/css/nodes.css
assets/css/edges.css
assets/css/drawers.css
assets/css/responsive.css
```

預期收益：

- 修改搜尋列、toolbar、節點卡片時，可以直接找到對應 CSS。
- 降低 `flowchart.html` 體積。
- 不大幅改動 JavaScript 行為，風險較低。

建議第一個可拆項目是 `flow-nav.css`，因為最近正在調整搜尋列與搜尋按鈕，範圍清楚且容易驗證。

## 第三階段：拆 JavaScript，使用過渡式 App 物件

不要一開始就全面改成複雜模組架構。比較穩定的做法是先保留目前頁面可運作的模式，逐步把 JS 拆成檔案。

建議初始結構：

```text
assets/js/config.js
assets/js/state.js
assets/js/storage.js
assets/js/flow.js
assets/js/search.js
assets/js/node-renderer.js
assets/js/edge-router.js
assets/js/selection.js
assets/js/import-export.js
assets/js/toolbar.js
assets/js/main.js
```

初期可以使用：

```text
window.FlowEditor
```

或單一 `App` 物件作為過渡層，避免一次引入太多模組化成本。等功能穩定後，再考慮改成真正的 ES modules。

## 第四階段：分離資料與畫面

目前許多函式同時做多件事，例如：

- 修改 `state`。
- 操作 DOM。
- 存檔。
- 重畫 edge。
- 顯示 toast。

後續應逐步收斂成以下角色：

- action：只負責修改資料。
- render：根據資料更新畫面。
- storage：負責儲存與讀取。
- ui event：只負責接事件並呼叫 action。

這樣未來改搜尋、節點、連線或儲存時，影響範圍會明顯縮小。

## 第五階段：建立 LLM 工作規則

建議新增：

```text
AGENTS.md
```

內容描述 Codex/LLM 在這個專案中應遵守的規則：

- 改搜尋，只看 `search.js` 與 `flow-nav.css`。
- 改節點樣式，只看 `nodes.css` 與 `node-renderer.js`。
- 改連線路由，只看 `edge-router.js`。
- 不要隨意新增全域變數。
- 新功能要先找對應模組，不直接塞進 `main.js`。
- DOM id/class 命名需要延續既有規則。
- 儲存格式不可任意破壞。

這份文件會讓後續使用 LLM 修改功能時，有更穩定的邊界與共同語言。

## 第六階段：逐步驗證

不要做單次大型重構。建議拆分順序如下：

1. 抽 CSS，確認畫面不變。
2. 抽 search JS，確認搜尋、上一筆、下一筆正常。
3. 抽 storage/API，確認讀寫專案正常。
4. 抽 node renderer，確認新增、編輯、tag、shape 正常。
5. 抽 edge router，確認連線、拖拉、重接正常。
6. 抽 import/export，確認 JSON/Mermaid 正常。

每一步完成後都應重新載入頁面並測試對應功能，確保問題能被限制在當次拆分範圍內。

## 建議第一刀

建議從以下兩件事開始：

1. 把搜尋列相關 CSS 抽到 `assets/css/flow-nav.css`。
2. 把搜尋相關 JavaScript 抽到 `assets/js/search.js`。

理由：

- 最近已經在修改搜尋 UI，脈絡清楚。
- 搜尋功能範圍相對獨立。
- 可以建立第一個可複製的拆分模式。
- 不需要先碰連線演算法、儲存格式或大型狀態管理。

這會是整個專案從單檔工具轉向可維護工具的第一個穩定切入點。
