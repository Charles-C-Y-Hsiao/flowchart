# Docs 導航文件維護流程

## 背景

專案目前開始建立 `docs/architecture.md` 與 `docs/feature-map.md`，用來幫助人與 LLM agents 在修改功能時快速定位 context。這兩份文件不應該寫成完整教學，而是作為導航工具：說明 module 邊界、state ownership、entry points、verify steps，以及哪些區域預設不需要讀取。

## 文件語言規則

- 程式相關名稱使用英文，例如 `Module`、`State`、`Entry points`、`Verify`、`Search Bar`、`state.search`、`flowchart.html`。
- 說明文字使用繁體中文，讓文件更貼近日常維護語境。
- Code symbols、file paths、CSS classes、DOM ids、function names 保持原樣。
- 不複製過多 implementation details，只記錄未來定位功能時需要知道的資訊。

## 每次 LLM 改完程式後的順手流程

可以把這當成固定收工檢查：

```text
改功能前：
1. 先讀 docs/architecture.md 判斷 module。
2. 再讀 docs/feature-map.md 找 feature entry points。
3. 只載入相關 context 開始改。

改功能後：
4. 確認這次是否改到 id / class / function / state / dependency。
5. 如果 entry point、verify steps 或 user-facing control 改變，更新 docs/feature-map.md。
6. 如果 module ownership、state ownership 或 dependency direction 改變，更新 docs/architecture.md。
7. Final handoff 說明 docs 已同步，或說明這次沒有 navigation boundary 變更所以不需更新。
```

## 可直接使用的提示詞

```text
請依照 docs/architecture.md 和 docs/feature-map.md 修改這個功能。
完成後，如果入口、state、模組責任或依賴有變，請同步更新這兩份 docs；
如果不用更新，請在最後說明原因。
```

## 目標

這個流程的目標不是讓每次修改都變成文件工作，而是讓文件保持輕、準、可導航。只要新的改動會影響下一次「從哪裡開始找」，就更新 docs；如果只是 function 內部細節，通常不需要更新。
