# 功能驗證步驟與命名對應

## 目的

這份文件整理一套可重複使用的 feature 驗證流程。未來驗證任何功能時，都可以照這個步驟檢查 `docs/architecture.md`、`docs/feature-map.md`、`docs/data-format.md`、`docs/api.md`、實際程式與 browser / API 行為是否一致。

同時也說明 `architecture.md` 的 module name 和 `feature-map.md` 的 feature name 不完全一樣時，是否會影響 LLM 判斷。

## 文件分工

```text
docs/architecture.md
→ 穩定的 module ownership / 邊界 / 依賴。

docs/feature-map.md
→ 穩定的 feature entry points / 驗證方式 / 導航入口。

docs/data-format.md
→ 穩定資料格式、JSON schema、storage payload、import/export 欄位。

docs/api.md
→ server routes、request / response、query params、persistence 行為。

docs/verification-status.md
→ 哪些 feature 已驗證、何時驗證、結果、是否補 docs、下一步。
```

`architecture.md` 和 `feature-map.md` 是導航文件，應盡量穩定；`verification-status.md` 是驗證看板，會隨著每次驗證更新。

## 核心原則：用 symbol 找入口，不要相信行號永遠正確

`feature-map.md` 可以保留 line number，因為它能幫助第一次閱讀時快速跳到附近；但 line number 不是穩定 contract。

真正穩定的入口是：

- function name，例如 `renderNode()`、`groupBounds()`、`layoutToolbarOverflow()`。
- CSS class，例如 `.node-group`、`.toolbar-more`。
- DOM id，例如 `#btnIO`、`#btnHelpToggle`。
- state key，例如 `state.display`、`state.groups`。
- API route，例如 `GET /api/flowchart`。
- data field，例如 `nodes.subflowId`、`display.flowchartTheme`。

如果只是前面新增幾行造成 line number 漂移，不需要重算整份文件。只有入口 symbol、責任範圍、檔案位置、API 或資料格式改變時，才需要更新 docs。

# 六步驗證法

將 "Prompt Guide" 照以下順序驗證：

```text
1. architecture.md 能判斷 module 邊界
2. feature-map.md 能找到 HTML / CSS / JS / state / API / data 入口
3. Select-String 搜尋結果與 docs 一致
4. Browser / API / data 操作正常
5. 讀完相關程式後沒有明顯缺漏
6. docs 有需要補的地方已更新
```

完成後再更新：

```text
7. docs/verification-status.md 已記錄驗證狀態
```

## 第 1 步：判斷 module 邊界

先看 `docs/architecture.md` 的 `Module Map`。

要確認：

- 這個功能屬於哪個主要 module。
- 這個 module 負責什麼。
- 它 owns / writes 哪些 state。
- 它 reads / depends on 哪些東西。
- 哪些區域預設不應該碰。

範例：

```text
Search Bar -> Search module
Account / Project Switchers -> Account / Project module
Storage / API / Persistence -> Storage / API module
Node Create / Edit / Render -> Node Renderer module
Toolbar / UI Shell -> Toolbar / UI Shell module
```

## 第 2 步：找到 feature entry points

再看 `docs/feature-map.md` 對應的 feature 章節。

要確認是否有：

- 使用者介面元素。
- 對應 modules。
- 目前程式入口。
- 主要 function。
- state / storage / API / data-format 入口。
- event binding。
- 驗證方式。
- 不建議從哪裡開始處理。

`feature-map.md` 是從使用者功能角度整理，所以章節名稱可以比 module 更具體。

## `feature-map.md` 建議固定格式

```md
## Feature Name

使用者介面元素：

- ...

對應 modules：

- 主要 module：`Primary Module`
- 相關 modules：`Related Module A`, `Related Module B`

目前程式入口：

- CSS / HTML / DOM / state / function / API / data entry points。

驗證方式：

- Browser / API / data / Select-String 驗證步驟。
- 不建議從哪裡開始處理。
```

### 為什麼要加「對應 modules」

`feature-map.md` 是從「功能」角度切；`architecture.md` 是從「module ownership」角度切。

像 `Node Renderer`、`Group Renderer`、`Toolbar / UI Shell` 這類 feature，本來就可能跨到互動、資料格式、流程範圍、儲存、drawer、prompt、edge redraw。

範例：

```md
對應 modules：

- 主要 module：`Toolbar / UI Shell`
- 相關 modules：`Display / Theme`, `Import / Export`, `Prompt Guide`, `Account / Project`
```

這樣 LLM 讀 feature 時會先知道：

- 主要應該修改哪個 module。
- 哪些周邊 module 只需要理解或驗證。
- 哪些地方不應該直接亂改。

## 第 3 步：用 Select-String 驗證

使用 PowerShell：

```powershell
Select-String -Path .\flowchart.html -Pattern 'keyword|functionName|className|stateName'
```

如果 feature 牽涉 server：

```powershell
Select-String -Path .\server.js -Pattern 'apiRoute|functionName|project|user'
```

如果 feature 牽涉 docs：

```powershell
Select-String -Path .\docs\*.md -Pattern 'Feature Name|Module Name|fieldName|apiRoute'
```

如果搜尋結果和 `feature-map.md` 對不上，就代表 docs 可能過時或不夠完整。

## 第 4 步：Browser / API / data 操作驗證

依 feature 風險決定驗證方式。

低風險功能可以直接操作：

- 搜尋。
- 打開 menu。
- 打開 drawer。
- 切換 view。

高風險功能先做只讀驗證：

- API GET。
- 讀 DOM state。
- 打開 menu 但不建立、不刪除、不切換。
- 讀 JSON data file 但不寫入。

例如 `Toolbar / UI Shell` 驗證時，可以：

- 確認 toolbar buttons 存在。
- 打開 More / help panel。
- 打開 Import/export drawer。
- 打開 Prompt drawer。
- 確認 drawer 互斥開關正常。
- 不先新增 node、不刪除、不套用匯入。

## 第 5 步：讀程式並整理簡短描述

讀完主要 function 後，整理一張表：

```text
函式 / 程式 | 簡短描述
```

描述保持短，目標是讓下一次讀 feature 的人知道每個 function 的角色。

如果發現 helper function 很重要，但 `feature-map.md` 沒列，就進入第 6 步補文件。

## 第 6 步：補 docs

需要補 docs 的情況：

- `feature-map.md` 漏掉重要 helper function。
- `feature-map.md` 沒有列 server/API 入口。
- `feature-map.md` 沒有列 data-format 入口。
- `architecture.md` 的 module ownership 不準。
- `data-format.md` 漏掉資料欄位。
- `api.md` 漏掉 endpoint 或 request / response 行為。
- 驗證時發現 feature 依賴其他 module。
- Browser / API 驗證步驟和 docs 不一致。

不需要補 docs 的情況：

- 只是 function 內部小改，不影響未來從哪裡找。
- 只是文字、樣式微調，entry point 沒變。
- 只是 line number 漂移，但 symbol 仍可搜尋且責任範圍沒變。

## 第 7 步：更新 verification-status.md

每次完成 feature 驗證後，更新 `docs/verification-status.md`：

```text
Feature
Primary Module
Related Modules
Status
Last Verified
Browser/API
Docs Updated
Notes
```

狀態建議：

```text
Not Started
In Progress
Done
Partial
Needs Docs Update
Blocked
```

不要把驗證狀態塞進 `architecture.md` 或 `feature-map.md`，避免導航文件被歷史紀錄污染。
